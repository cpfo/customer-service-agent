# 项目架构与设计

## 概述

基于 Spring AI Alibaba + DashScope (通义千问) 的智能客服 AI Agent，通过 RAG（检索增强生成）技术回答电商客服场景问题。

## 技术栈

| 组件 | 选型 | 版本 |
|------|------|------|
| 开发语言 | Java | 17 |
| 框架 | Spring Boot + Spring AI | 3.3.13 / 1.1.2 |
| AI 平台 | Spring AI Alibaba (DashScope) | 1.1.2.2 |
| 大语言模型 | qwen-flash | — |
| 嵌入模型 | text-embedding-v3 | — |
| 重排序模型 | gte-rerank (DashScope) | — |
| 向量库 | Redis Stack (RediSearch) | 7.2 |
| API 文档 | SpringDoc OpenAPI (Swagger UI) | 2.6.0 |
| 构建工具 | Maven (Wrapper 3.9.9) | 兼容 3.3.9 |
| 编码 | UTF-8 (全链路强制) | — |

```
用户请求
    │
    ├── GET /api/chat                        ── 同步文本返回
    │       │
    │       ▼
    └── GET /api/chat/stream                 ── SSE 流式返回
            │
            ▼
     buildRAGContext()  (共同逻辑)
            │
            ├── 1. 查询改写 (QueryRewriter)
            │      LLM 改写 → [原始, 改写1, 改写2]
            │
            ├── 2. 向量检索
            │      RedisVectorStore.topK(vectorTopK)
            │
            ├── 3. 重排序 (ReRankerService)
            │      ├── 交叉编码器模式: DashScopeReranker (gte-rerank)
            │      └── Hybrid 模式: 向量70% + 关键词30%
            │
            ├── 4. 构建 System Prompt        ── 有/无上下文
            ├── 5. 上下文压缩 (ContextCompressor)
            │      LLM 摘要压缩历史 → 或原始消息
            │
            │
            ▼

            │
            ├── 同步: .call().content()      ── 返回 String
            │
            └── 流式: .stream().content()    ── Flux → SseEmitter
                          │
                          ├── data: {"content":"..."}
                          ├── event: done
                          └── event: error
            │
            ▼
     SessionService.saveExchange()  ── Redis 持久化
```

## 核心组件说明

### 1. `CustomerServiceAgentApplication` — 启动入口
标准 Spring Boot 启动类，无额外配置。

### 2. `RAGConfig` — 配置层
- `RedisVectorStore`：Redis Stack 向量库，基于 RediSearch 模块实现向量相似度搜索
- `VectorStore` 接口 Bean：指向同一 `RedisVectorStore` 实例，供检索使用
- `ChatClient`：Spring AI 对话客户端，无默认 system prompt（由 Controller 动态传入）

### 3. `OpenAPIConfig` — API 文档配置
- 通过 `springdoc-openapi-starter-webmvc-ui:2.6.0` 自动配置 Swagger UI
- `@OpenAPIDefinition` 注解声明 API 标题、版本、联系方式、许可证
- 配合 `@Tag` / `@Operation` / `@Parameter` / `@ApiResponse` 注解，生成结构化的交互式 API 文档
- 端点地址：
  - **Swagger UI**：`http://localhost:8080/swagger-ui.html`
  - **OpenAPI JSON**：`http://localhost:8080/v3/api-docs`
  - **OpenAPI YAML**：`http://localhost:8080/v3/api-docs.yaml`
- 无需额外配置类（starter 自动注册 `/v3/api-docs` 和 `/swagger-ui/*` 端点）
- `OpenAPIConfig` 仅提供文档元信息，不涉及业务逻辑

### 4. `KnowledgeService` — 知识库生命周期管理
- `CommandLineRunner`：启动时自动执行
- 加载策略：检查 Redis 中 `app:knowledge:initialized` 标记是否存在
  - 已标记 → 跳过（避免重复加载）
  - 未标记 → 扫描 `classpath:knowledge/*.{txt,pdf,docx,md,html}` → `TikaDocumentReader` 读取 (Apache Tika 自动检测格式) → `TokenTextSplitter` 分块 → `RedisVectorStore.add()` 写入 → 设置标记
- Redis 自身持久化，应用无需 save/load 逻辑

### 5. `ReRankerService` — 重排序
- 支持两种模式，通过 `app.reranker.type` 切换：

  **交叉编码器模式**（推荐，默认）：
  - 调用 `DashScopeReranker` → DashScope `gte-rerank` API
  - 对 query 与每篇文档做联合评分，精度 +15%~30%
  - 失败时自动降级返回原顺序

  **混合模式**：
  - 输入：$TopK_{retrieve}$ 条原始检索结果
  - 评分公式：$\text{score} = \text{vectorScore} \times 0.7 + \text{keywordOverlap} \times 0.3$
  - `vectorScore`：`Document.getScore()`（向量库返回的相似度）
  - `keywordOverlap`：问题与文档的公共词占比

- 输出：$TopK_{rerank}$ 条（默认 3）

### 6. `QueryRewriter` — 查询改写
- 调用 LLM 将用户口语问题改写为多条精炼书面查询（默认 2 条）
- 改写侧重不同关键词角度，去除口语化表达
- 改写结果与原始查询合并去重，后续向量检索对每条改写分别匹配
- 失败时自动降级使用原始查询

### 7. `DashScopeReranker` — 交叉编码器重排序
- 通过 HTTP 调用 DashScope `gte-rerank` 模型的 rerank API
- 将 query 与每篇文档配对，输出 0~1 相关性分数
- 与 `ReRankerService` 集成，作为其交叉编码器模式的底层引擎
- 失败时 `ReRankerService` 自动降级

### 8. `ContextCompressor` — 上下文压缩
- 当历史消息超过 4 条时，调用 LLM 压缩为精炼摘要
- 保留用户意图、已解决问题、待办事项等关键信息
- 压缩结果以 `【对话历史摘要】` 注入 system prompt
- 短对话或 LLM 失败时降级为原始消息拼接

### 9. `DocumentStore` — 文档缓存
- 纯内存缓存：`KnowledgeService` 加载文档后同步写入
- 提供 `addAll()` / `getAll()` / `clear()` 方法，供文件内容预览和删除操作使用

### 10. `SessionService` — 会话历史管理
- 使用 `StringRedisTemplate` + Jackson 序列化，将消息列表存为 JSON
- 存储结构：Redis key `chat:session:{sessionId}` → value `[{"type":"user","content":"..."}, ...]`
- 自动截断：保留最近 10 轮对话（20 条消息），防止历史过长
- 反序列化失败时自动重置会话

### 11. `ChatController` — 请求入口与 RAG 编排
- 两个端点共享 `buildRAGContext()` 提取的共同逻辑（检索 + 重排序 + 构造 prompt + 获取历史）

#### 同步端点 `GET /api/chat`
- 返回 `text/plain`，`chatClient.prompt().call().content()` 一次返回完整回复
- 兼容旧版客户端，适合简单集成

#### 流式端点 `GET /api/chat/stream`
- 返回 `text/event-stream`，使用 `SseEmitter` + `Flux<String>` 逐 token 推送
- SSE 事件格式：
  - `data: {"content":"token文本"}` — 每个 token
  - `event: done` — 流结束
  - `event: error` — 异常
- 首字延迟 ~200ms

#### 共同流程
1. 检索（SearchRequest.topK = vectorTopK）
2. 重排序（ReRankerService）
3. 构造 system prompt（有/无上下文两套模板）
4. 注入历史（SessionService.getHistory，最近 10 轮）
5. 注入工具（CustomerTools → `.tools()`）
6. 调用 LLM → 返回回复
7. 保存 Exchange 到会话（SessionService.saveExchange）
- 会话管理：委托 `SessionService`，Redis 持久化，重启不丢失

### 12. `CustomerTools` — Function Calling 工具
- 三个 `@Tool` 方法，模型根据用户意图自动选择调用：
  - `getOrderStatus(orderId)` — 查询订单状态
  - `trackLogistics(orderId)` — 查询物流信息
  - `submitRefundRequest(orderId, reason)` — 提交退货申请
- 使用 `@Tool(description=...)` + `@ToolParam(description=...)` 注解，模型自动理解参数含义
- 当前为 Mock 数据，后续可对接真实业务 API

## 配置参考

```yaml
# application.yml
spring:
  ai:
    dashscope:
      api-key: ${AI_DASHSCOPE_API_KEY}      # API 密钥 (环境变量)
      chat:
        options:
          model: qwen-flash                   # 大模型
      embedding:
        options:
          model: text-embedding-v3            # 嵌入模型
  redis:
    host: localhost                            # Redis 主机
    port: 6379                                 # Redis 端口

app:
  retrieval:
    vector-top-k: 10                           # 向量检索条数
    fusion-top-k: 10                           # 重排序输入条数

  reranker:
    type: cross-encoder                        # hybrid | cross-encoder
    top-k: 3                                   # 最终保留文档数
    cross-encoder-model: gte-rerank            # DashScope rerank 模型

  query-rewriting:
    enabled: true                              # 启用查询改写
    num-rewrites: 2                            # 改写数量

  context-compression:
    enabled: true                              # 启用上下文压缩
```
> **API 文档**：启动应用后可通过 `http://localhost:8080/swagger-ui.html` 查看在线文档并调试接口；OpenAPI 原始规范位于 `http://localhost:8080/v3/api-docs`（JSON）或 `http://localhost:8080/v3/api-docs.yaml`（YAML）。

## MCP Server — 用户管理模块 (`spring-mcp-server`)

### 概述

独立 MCP Server 进程（端口 **8081**），提供用户管理功能，通过 **Streamable HTTP MCP 协议** 暴露工具，供 Agent 端 (`customer-service-agent`) 的 LLM 动态调用。

### 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| MCP 服务端 | `spring-ai-starter-mcp-server-webmvc` | Spring AI MCP Server (WebMVC SSE/Streamable) |
| HTTP | `spring-boot-starter-web` | Servlet 容器 |
| 数据存储 | `ConcurrentHashMap` (内存) | 无数据库依赖，重启数据丢失，适合演示 |
| 协议 | Streamable HTTP | MCP 传输层，`/mcp` 端点 |

### 暴露的 MCP Tool（6 个）

| Tool 方法 | 功能 | 输入 | 输出 |
|-----------|------|------|------|
| `listUsers` | 获取所有用户列表 | 无 | `List<User>` |
| `getUserById` | 根据 ID 查询用户 | `id` | `User` 或 null |
| `createUser` | 创建新用户 | `name`, `email` | `User` |
| `updateUser` | 更新用户信息 | `id`, `name?`, `email?`, `phone?` | `User` 或 null |
| `deleteUser` | 删除用户 | `id` | `boolean` |
| `searchUsers` | 按关键字搜索用户 | `keyword` | `List<User>` |

### 请求链路（Agent → MCP Server）

```
ChatController
  └── ChatClient
        └── DashScopeChatModel (qwen-flash)
              └── DefaultToolCallingManager
                    └── SyncMcpToolCallback          ← spring-ai-starter-mcp-client

                                └── UserMcpTools (spring-mcp-server:8081/mcp)
                                      └── UserService
                                            └── UserRepository (ConcurrentHashMap)

```

### 配置

**Agent 端** (`customer-service-agent/application.yml`):
```yaml

    mcp:
      client:
        type: sync
        streamable-http:
          connections:
            fastapi-mcp-server:         # 连接名称
              url: http://localhost:8081
              endpoint: /mcp
```

    mcp:

```
