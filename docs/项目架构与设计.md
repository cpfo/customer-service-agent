# 项目架构与设计

## 概述

基于 Spring AI Alibaba + DashScope (通义千问) 的智能客服 AI Agent，通过 RAG（检索增强生成）技术回答电商客服场景问题。

## 技术栈

| 组件 | 选型 | 版本 |
|------|------|------|
| 开发语言 | Java | 17 |
| 框架 | Spring Boot + Spring AI | 3.3.13 / 1.1.2 |
| AI 平台 | Spring AI Alibaba (DashScope) | 1.1.2.2 |
| 大语言模型 | qwen-max | — |
| 嵌入模型 | text-embedding-v3 | — |
| 向量库 | Redis Stack (RediSearch) | 7.2 |
| 构建工具 | Maven (Wrapper 3.9.9) | 兼容 3.3.9 |
| 编码 | UTF-8 (全链路强制) | — |

## 数据流

```
用户请求
    │
    ▼
ChatController.chat(sessionId, message)
    │
    ├── 1. VectorStore.similaritySearch()     ── 向量检索
    │       │                                    (SearchRequest.topK=10)
    │       ▼
    ├── 2. ReRankerService.reRank()           ── 重排序
    │       │                                    (向量70% + 关键词30%)
    │       ▼
    ├── 3. 构建 System Prompt                 ── 组装提示词
    │       │                                    (有上下文 / 无上下文)
    │       ▼
    ├── 4. ChatClient.prompt()                ── 调用 LLM
    │       │                                    + 注入对话历史(最近10轮)
    │       ▼
    ├── 5. 保存 Exchange 到会话              ── 持久化历史
    │
    ▼
返回回复给用户
```

## 核心组件说明

### 1. `CustomerServiceAgentApplication` — 启动入口
标准 Spring Boot 启动类，无额外配置。

### 2. `RAGConfig` — 配置层
- `RedisVectorStore`：Redis Stack 向量库，基于 RediSearch 模块实现向量相似度搜索
- `VectorStore` 接口 Bean：指向同一 `RedisVectorStore` 实例，供检索使用
- `ChatClient`：Spring AI 对话客户端，无默认 system prompt（由 Controller 动态传入）

### 3. `KnowledgeService` — 知识库生命周期管理
- `CommandLineRunner`：启动时自动执行
- 加载策略：检查 Redis 中 `app:knowledge:initialized` 标记是否存在
  - 已标记 → 跳过（避免重复加载）
  - 未标记 → 扫描 `classpath:knowledge/*.{txt,pdf,docx,md,html}` → `TikaDocumentReader` 读取 (Apache Tika 自动检测格式) → `TokenTextSplitter` 分块 → `RedisVectorStore.add()` 写入 → 设置标记
- Redis 自身持久化，应用无需 save/load 逻辑

### 4. `ReRankerService` — 重排序
- 输入：$TopK_{retrieve}$ 条原始检索结果
- 评分公式：
  $$\text{score} = \text{vectorScore} \times 0.7 + \text{keywordOverlap} \times 0.3$$
  - `vectorScore`：`Document.getScore()`（向量库返回的相似度）
  - `keywordOverlap`：问题与文档的公共词占比
- 输出：$TopK_{rerank}$ 条（默认 3）

### 5. `SessionService` — 会话历史管理
- 使用 `StringRedisTemplate` + Jackson 序列化，将消息列表存为 JSON
- 存储结构：Redis key `chat:session:{sessionId}` → value `[{"type":"user","content":"..."}, ...]`
- 自动截断：保留最近 10 轮对话（20 条消息），防止历史过长
- 反序列化失败时自动重置会话

### 6. `ChatController` — 请求入口与 RAG 编排
- REST API：`GET /api/chat?sessionId=xxx&message=xxx`
- 请求流程：
  1. 检索（SearchRequest.topK = retrieveTopK）
  2. 重排序（ReRankerService）
  3. 构造 system prompt（有/无上下文两套模板）
  4. 注入历史（SessionService.getHistory，最近 10 轮）
  5. 调用 LLM → 返回回复
  6. 保存 Exchange 到会话（SessionService.saveExchange）
- 会话管理：委托 `SessionService`，Redis 持久化，重启不丢失

## System Prompt 策略

### 有上下文模式
```
你叫"小智"，专业电商客服。
规则：基于知识回答，不编造，结合历史理解问题。
附：知识库相关内容
```

### 无上下文模式
```
你叫"小智"。
规则：优先用历史回答，否则引导转人工(400-888-8888)，不编造。
```

## 配置参考

```yaml
# application.yml
spring:
  ai:
    dashscope:
      api-key: ${AI_DASHSCOPE_API_KEY}      # API 密钥 (环境变量)
      chat:
        options:
          model: qwen-max                     # 大模型
      embedding:
        options:
          model: text-embedding-v3            # 嵌入模型
  redis:
    host: localhost                            # Redis 主机
    port: 6379                                 # Redis 端口

app:
  reranker:
    top-k: 3                                   # 最终保留文档数
    retrieve-top-k: 10                         # 向量库检索数
```

## 日志体系

| 级别 | 内容 |
|------|------|
| ERROR | 知识库加载失败、LLM 调用异常、会话历史保存失败 |
| WARN | 会话历史反序列化失败、重排序输入为空 |
| INFO | 请求开始/结束、消息内容、检索数、重排序得分、回复预览 |
| DEBUG | 知识全文、历史消息详情、system prompt 全文 |
| TRACE | 每条文档的评分明细 |

## Phase 路线图

| Phase | 内容 | 状态 |
|-------|------|------|
| Phase 1 | 基础架构 + RAG 检索 | ✅ 完成 |
| Phase 2 | 向量库持久化 + 重排序 + 多轮对话 | ✅ 完成 |
| Phase 3 | 生产化增强（Redis + 多格式文档） | ✅ 完成 |
| Phase 4 | 精度提升 | ⏳ 待定 |
| Phase 5 | 运维与管理 | ⏳ 待定 |

### Phase 3 — 生产化增强
| 改进项 | 当前 | Phase 3 结果 | 说明 |
|--------|------|-------------|------|
| 向量库 | SimpleVectorStore (内存+文件) | ✅ Redis Stack (RediSearch) | 专业向量数据库，数据持久化到磁盘，多实例共享 |
| 会话存储 | ConcurrentHashMap (内存) | ✅ Redis (StringRedisTemplate) | 重启不丢失，支持水平扩展 |
| 文档格式 | 仅 TXT | ✅ TXT/PDF/DOCX/MD/HTML (Apache Tika) | TikaDocumentReader 自动检测格式，零配置添加新文件 |
| 响应方式 | 同步 HTTP | ⏳ 待实现 | SSE 流式逐字返回 |
| Function Calling | 无 | ⏳ 待实现 | 让 Agent 能调用真实业务接口 |

### Phase 4 — 精度提升
| 改进项 | 当前 | 目标 | 说明 |
|--------|------|------|------|
| 重排序 | 向量分 + 关键词分 (混合) | 交叉编码器 (BGE-Reranker) | ONNX 本地运行，精度 +15%~30% |
| 查询改写 | 用户原句直接检索 | LLM 改写 → 多角度检索 → 融合 | 解决口语与书面语不匹配问题 |
| 召回策略 | 单一向量检索 | 向量 + BM25 多路召回 | Elasticsearch / 自定义 BM25 互补 |
| 上下文压缩 | 固定截取 10 轮 | LLM 摘要压缩历史 | 避免 Token 超限，保留关键信息 |

### Phase 5 — 运维与管理
| 改进项 | 说明 |
|--------|------|
| 知识库管理 API | POST/GET/DELETE `/api/knowledge` 动态管理文档，热更新向量库 |
| 可观测性 | Micrometer + Prometheus + Grafana，监控延迟 / Token 消耗 / 召回准确率 |
| 请求缓存 | 相同问题 + 相同上下文命中缓存，降低 API 成本 |
| 容错降级 | 模型超时 / 限频时返回缓存结果或引导转人工 |
| 人工转接 | 检测到低置信度时自动创建工单，转接人工客服 (400 热线 / 在线坐席) |

## 依赖关系

```
ChatController
  ├── ChatClient          ← RAGConfig
  ├── VectorStore         ← RAGConfig → RedisVectorStore
  ├── SessionService      ← 自定义 Service → StringRedisTemplate
  ├── ReRankerService     ← 自定义 Service

SessionService
  └── StringRedisTemplate ← spring-boot-starter-data-redis

KnowledgeService
  ├── VectorStore         ← RAGConfig → RedisVectorStore
  └── StringRedisTemplate ← spring-boot-starter-data-redis

RedisVectorStore
  ├── EmbeddingModel      ← DashScope text-embedding-v3
  └── JedisPooled         ← spring-ai-redis-store (内嵌)
```
